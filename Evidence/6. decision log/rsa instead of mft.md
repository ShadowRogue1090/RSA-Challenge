Originally I was planning to design a challenge around using the AnalyzeMFT python package that a user would need to run and then go digging through a master file table list of contents for a flag.

While I spent a few weeks researching this, I found it was going to be to impractical to get working for the constraints I had. Not only had I only used the script once myself in a controlled environment, but I was unsure how to even make the MFT I needed in the first place. I was re-reading through my notebook that I got the AnalyzeMFT idea from and round another subject I could talk about: RSA encryption.

RSA being an asymmetrical encryption method can sound confusing at first but the logic is quite simple once explained. As a quick test, I drew a quick diagram and explanation to Matt and Josh of what asymmetric (RSA) and symmetric (AES) encryption roughly works to see how it was. Being that they were unfamiliar with the concept, I decided it could be a better option to follow. Not only was I more confident with the logic, but I could easily visualise was a docker container challenge could look like. Have two containers, Alice and Bob, and put them on the same internal docker network. If one is running a python script with some if/else statements, and the other has netcat & ssh, then a user could log into one, and interact with a bot in the other. While this could be a single container with no network, I think having a network makes it more authentic.

One interesting option would be to have a third container on the network to listen into the conversation and record all the packets going inbetween with wireshark. Then if the user accidentally transmits something sensitive, like their private key, then the hacker could send an interrupting "YOU FAILED" message and end the challenge pre-maturely. I believe I need to use termshark for docker containers as we cannot run a GUI in this instance, but as long as there is a .pcap file and necessary filters, it should work.
